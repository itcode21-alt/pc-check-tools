# ITSVC 사이트 운영 실행 계획

## 목적과 원칙

- 목표는 광고·제휴 클릭 수가 아니라 사용자가 안전하게 다음 점검 또는 업그레이드 결정을 할 수 있게 돕는 것이다.
- 새 콘텐츠와 기능은 `정확성 → 사용자 흐름 → 수익화` 순서로 검토한다.
- AI 문의·로그·분석 데이터는 사용자의 명시 동의가 있을 때만 저장하며, 개인을 식별할 수 있는 정보는 콘텐츠 개선 자료로 사용하지 않는다.
- 모든 배포는 SSH를 기본으로 하며, **운영 URL 확인이 끝나기 전에는 작업 완료로 처리하지 않는다.**

## 1. 배포와 장애 대응

### 매 배포 작업

1. 변경 범위와 영향 페이지를 확인한다.
2. 자바스크립트 문법 검사, HTML 연결 링크, 생성 파일(검색 인덱스·사이트맵)을 점검한다.
3. Git 커밋 후 SSH 원격으로 `main` 브랜치에 푸시한다.
4. 정적 사이트 반영을 기다린 뒤 실제 `https://itsvc.co.kr/...` 주소에서 새 문구·스크립트·링크를 확인한다.
5. AI API가 바뀐 경우 AI 서버에서 최신 코드를 pull하고 서비스를 재시작한 뒤 `/api/health`와 새 API를 확인한다.
6. 배포 일시, 커밋 ID, 확인 URL, 결과를 배포 기록에 남긴다.

### 장애 발생 시

- 정적 페이지 오류: 마지막 정상 커밋을 확인하고 문제 파일만 되돌린 뒤 재배포한다.
- AI API 오류: `/api/health`, 서비스 로그, 환경 변수, CORS, 새 엔드포인트 순으로 확인한다.
- 쿠팡 링크 API 오류: 화면에서 제휴 링크를 숨기고 진단 결과는 계속 제공한다. API 키를 브라우저 코드에 넣지 않는다.
- 검색 인덱스 누락: `scripts/build-search-index.mjs`를 실행하고 생성된 `search-index.js`를 함께 배포한다.

## 2. 주간 운영 작업

### 매주 월요일: 품질 점검

- 상위 유입 페이지, 내부 링크, 404 페이지, 검색 실패어를 확인한다.
- AI 답변에서 근거가 부족한 항목과 사용자가 반복해서 질문한 증상을 분류한다.
- 새 오류코드·Windows 업데이트·게임 오류가 있으면 공식 출처부터 확보한다.

### 매주 수요일: 콘텐츠 보완

- 반복 문의 중 개인정보가 제거된 증상·오류코드·환경 정보만 집계한다.
- 기존 항목과 중복되는지 확인한 후, 기존 페이지 보완 또는 신규 페이지 중 하나를 선택한다.
- 신규 페이지는 `증상/상황 → 원인 후보 → 안전한 점검 순서 → 관련 도구 → 공식 출처` 구조로 작성한다.
- 부품 교체 내용은 호환성 확인 항목과 “고장 점검 우선” 조건을 함께 쓴다.

### 매주 금요일: 기능·수익화 점검

- 계산기 입력값·결과 문구·모바일 화면을 실제로 확인한다.
- 제휴 링크는 교체 필요 또는 사용자가 선택한 조건이 분명한 결과에만 노출한다.
- 링크 주변의 제휴 고지, 검색 결과 한계, 호환성 확인 문구가 표시되는지 확인한다.
- 클릭 수가 많아도 이탈률·오류 신고가 높은 링크는 중단하거나 조건을 좁힌다.

## 3. 월간 운영 작업

### 콘텐츠와 SEO

- Search Console에서 상위 검색어, 노출은 많지만 클릭이 낮은 페이지, 색인 오류를 확인한다.
- 상위 10개 페이지의 제목, 설명, 내부 링크, 최종 검토일, 공식 출처를 갱신한다.
- `sitemap.xml`, 화면 사이트맵, 도구 목록, 검색 인덱스를 신규 페이지와 맞춘다.
- 뉴스는 월간 Windows 업데이트 이슈와 영향이 큰 게임·드라이버 문제만 발행한다.

### 데이터와 개인정보

- 저장 동의 문의는 보관 기간이 지난 원문을 삭제하고, 통계만 남길지 검토한다.
- 문의 데이터에 계정·전화번호·주소·라이선스 키·전체 로그 등 민감한 내용이 섞였는지 점검한다.
- 개인정보처리방침, 동의 문구, 저장 목적이 실제 동작과 일치하는지 확인한다.

### 성과 검토

- 측정 지표: 검색 유입, 계산기 실행 수, AI 진단 완료 수, 관련 가이드 이동률, 제휴 링크 클릭률, 오류 신고 수.
- 좋은 지표: 사용자가 진단 뒤 관련 가이드나 계산기로 이동하고, 반복 검색 없이 다음 행동을 선택하는 비율.
- 나쁜 지표: 제휴 링크 클릭만 높고 가이드 이탈·문의 불만이 함께 증가하는 경우.

## 4. 분기 운영 작업

- 중복·짧은 페이지를 통합하고 오래된 내부 링크를 정리한다.
- 오류코드·이벤트·증상·게임 데이터의 연결이 실제로 맞는지 표본 검토한다.
- 모바일 접근성: 글자 대비, 카드 여백, 모달, 키보드 초점, 버튼 크기를 전수 확인한다.
- AI 답변 실패 유형을 `근거 부족`, `검색 실패`, `질문 모호`, `새 콘텐츠 필요`로 분류하고 개선 순서를 정한다.
- 제휴 링크와 광고가 진단 흐름을 방해하는 위치에 없는지 검토한다.

## 5. 기능 개발 우선순위

1. AI 서버 배포 자동화와 상태 확인: 정적 페이지와 API의 배포 확인 절차를 분리한다.
2. 업그레이드 영역: RAM 가능 여부, SSD 교체, 그래픽카드·파워 호환성, 노트북 제한 사항을 연결한다.
3. 계산기 개선: 실제 장착 파워 비교, SSD 선택 조건 기반 추정, 백업 용량 계산처럼 입력 근거가 명확한 기능만 만든다.
4. 사용자 피드백: AI 답변·계산기 결과의 도움 여부를 수집하되 개인정보 저장 동의와 분리한다.
5. 운영 화면: 검색 실패어, 반복 문의, 링크 오류, API 상태를 빠르게 보는 내부 점검 목록을 만든다.

## 6. 새 콘텐츠 또는 계산기 출시 체크리스트

- 기존 페이지·계산기와 중복되지 않는가?
- 입력값이 일반 사용자가 실제로 확인할 수 있는 정보인가?
- 결과가 확정 진단 또는 무조건 구매 권유로 보이지 않는가?
- 기기 모델·부품별 차이를 확인하는 방법이 안내되어 있는가?
- 모바일에서 글자·버튼·카드 여백이 읽기 쉬운가?
- 검색 인덱스, 도구 목록, 화면/XML 사이트맵, 관련 가이드 링크를 추가했는가?
- 운영 URL에서 실제 반영을 확인했는가?
