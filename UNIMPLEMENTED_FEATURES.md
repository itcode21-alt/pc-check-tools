## 1. AI 서버 배포 자동화 및 상태 확인 (높음)

### 현재 상태
- AI 서버(`ai-service/app.py`)는 Mac mini에서 수동으로 실행 중
- 배포 스크립트 `ai-service/deploy.sh`가 존재하나, 자동화 여부 미확인
- `/api/health` 엔드포인트: 서비스 상태를 확인할 수 있음

### 미구현 작업
- [ ] AI 서버 자동 시작/재시작 (launchd 설정 존재 - `ai-service/launchd/kr.co.itsvc.ai-service.plist`)
- [ ] 배포 파이프라인 자동화: Git push → AI 서버 자동 pull & restart
- [ ] 모니터링 대시보드: `/api/health` 상태를 주기적으로 체크하고 장애 알림
- [ ] 정적 페이지와 AI 서버의 배포 확인 절차 분리
  - 현재: 사람이 수동으로 두 곳 모두 확인
  - 목표: 자동화된 상태 확인 + 실패 시 자동 롤백

### 추천 구현 방향
1. GitHub Actions 워크플로우 작성
   - `ai-service/` 폴더 변경 감지 → 자동 배포
   - `/api/health` 체크 통과 시에만 정적 페이지 배포 진행
2. 내부 관리 화면에 최근 배포 상태 표시
3. Slack/이메일 알림 연동

## 2. 업그레이드 영역 간 연결 (높음)

### 현재 상태
- 개별 계산기 존재: RAM 업그레이드 체커, SSD TBW 계산기, 파워/전원 계산기, 백업 스토리지 계산기 등
- 각 계산기가 독립적으로 동작
- 업그레이드 가능 여부를 종합적으로 판단하는 페이지 없음

### 미구현 작업
- [ ] **"내 PC 업그레이드 가능 영역" 통합 진단**
  - 입력값: 제조사, 모델명/사양(CPU, GPU, RAM, SSD 현황)
  - 출력: 각 영역별 업그레이드 가능 여부
    - RAM: 슬롯 수, 최대 용량, 호환 메모리 타입
    - SSD: M.2 슬롯 수, 규격(NVMe/SATA), 인터페이스
    - 그래픽카드: 슬롯 호환성, 전력 연결 가능 여부
    - 파워: 현재 와트, 업그레이드 필요 와트
  - 디바이스별 제약사항 (노트북: 파워 교체 불가 등)

- [ ] **부품 호환성 데이터베이스**
  - `data.js`에 `parts` 배열이 있으나, 호환성 매트릭스 부재
  - 필요: CPU ↔ 메인보드 호환성, 그래픽카드 ↔ 파워 호환성

- [ ] **페이지 간 링크 개선**
  - 진단 페이지에서 "업그레이드 가능" 판정 후 → 계산기로 이동
  - 계산기 결과에서 부품 추천 → Coupang 제휴 링크로 유도

### 데이터 구조 예시
```js
// data.js 추가 필요
{
  id: "dell-xps-13-9370",
  manufacturer: "Dell",
  model: "XPS 13 9370",
  specs: {
    cpu: "Intel 8th Gen", 
    maxRAM: 32, // GB
    ramSlots: 2,
    nvmeSlots: 1,
    canUpgradeGPU: false, // 노트북 제약
    canUpgradePower: false,
  },
  powerConnectors: ["8-pin"] // 그래픽카드 전력 연결
}
```

## 3. 계산기 개선 (중간)

### 현재 상태
- PSU(파워) 계산기: 실행됨
- RAM 업그레이드 체커: 실행됨
- SSD TBW 계산기: 실행됨
- 백업 스토리지 계산기: 실행됨
- RAID 계산기: 실행됨
- 모니터 계산기: 실행됨

### 미구현 작업
- [ ] **실제 장착 파워와의 비교 기능**
  - 현재: "몇 와트가 필요한가?"만 답변
  - 개선: 사용자의 현재 파워 와트 입력 → "부족한가?"를 직접 알려줌

- [ ] **SSD 선택 조건 기반 추정**
  - 현재: TBW만 계산
  - 개선: 
    - 게임 설치 용량 선택
    - 영상 편집 여부
    - 업무 문서 백업 필요도
    → 추천 용량을 동적으로 계산

- [ ] **백업 용량 계산 고도화**
  - 현재: 단순 배수(1배, 2배, 3배) 선택
  - 개선:
    - 변경 빈도 입력 (일일/주간/월간)
    - 보관 기간 (6개월, 1년, 3년)
    - → 실제 필요 용량 역산

- [ ] **모바일 반응형 개선**
  - 현재: 데스크톱 기준으로 설계됨
  - 개선: 스마트폰에서의 입력 UX 개선

### 우선순위 내에서의 순서
1. 실제 장착 파워 비교 (사용자가 가장 궁금해함)
2. SSD 용량 추정 개선 (게임/작업 기반)
3. 백업 용량 역산

## 4. 사용자 피드백 시스템 (중간)

### 현재 상태
- 개별 페이지나 AI 진단 결과 후 "이 답변이 도움이 되었나요?" 여부는 `inquiry_store.py`에 저장될 수 있음
- 하지만 사용자의 선택 동의와 분리되어 있지 않음 (개인정보 저장 동의 없이 수집되는 문제)

### 미구현 작업
- [ ] **진단 및 계산 결과 후 피드백 수집**
  - UI: 진단 결과 하단에 "이 진단이 정확했나요?" 버튼
  - 저장: `inquiry_id` + 사용자 피드백만 저장 (개인정보 없음)
  - 분석: 어떤 진단이 자주 부정적 평가를 받는지 월간 리뷰

- [ ] **계산기 결과 검증**
  - UI: "내 파워는 이 권장값과 달라요" → 피드백 폼
  - 목적: 실제 하드웨어와 계산 결과의 간격 파악
  - 예: "RTX 4090 + Ryzen 5 조합인데 800W로도 잘 돈다"

- [ ] **명시적 동의 분리**
  - 피드백: 선택적 (즉시 제공)
  - 개인정보 저장: 별도 체크박스 (현재는 무조건 저장되고 있을 수 있음)

### 데이터 보호 원칙
- 피드백 원문은 6개월 후 자동 삭제
- 통계만 영구 보관
- IP/사용자 식별 정보 절대 저장 금지

- 데이터: 기존 로그 + 주기적 집계

## 구현 순서 권장안

### Phase 1 (1~2주) - 긴급성 높음
1. **AI 서버 배포 자동화** 
   - GitHub Actions 워크플로우 작성
   - `/api/health` 자동 체크
   - 영향도: 현재 수동 배포 시간 단축, 안정성 향상

### Phase 2 (2~4주) - 사용성 중간
2. **업그레이드 영역 통합 진단**
   - 부품 호환성 데이터 구성
   - 통합 진단 로직 (app.js 확장)
   - 영향도: 사용자의 구매 결정 신뢰도 향상

3. **사용자 피드백 시스템**
   - 동의 분리 + 피드백 수집
   - 월간 리뷰 프로세스 수립
   - 영향도: 콘텐츠 품질 개선 데이터 확보

### Phase 3 (4~8주) - 운영 효율성
4. **계산기 개선**
   - 실제 파워 비교 기능
   - SSD 용량 추정 고도화
   - 영향도: 사용자 경험 개선, 재방문율 증가

5. **운영 대시보드**
   - 기본 모니터링 페이지
   - SEO/수익 통계
   - 영향도: 운영 시간 단축, 데이터 기반 의사결정
